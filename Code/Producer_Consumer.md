``` c
// Producer
while (true) {
    /* 1. 새로운 아이템을 생산하여 'next_produced' 변수에 임시 저장한다. */
    
    /* 2. 버퍼가 가득 찼는지 검사한다. */
    // counter가 버퍼의 최대 크기(BUFFER_SIZE)와 같다면 더 이상 넣을 공간이 없으므로,
    // 공간이 생길 때까지 무한 루프를 돌며 대기(Busy Waiting)한다.
    // Busy waiting은 CPU 효율성이 굉장히 안좋은 방식이다.
    while (counter == BUFFER_SIZE) ; /* do nothing */
    
    /* 3. 버퍼에 데이터를 삽입한다. */
    // 현재 'in'이 가리키는 인덱스 위치에 생산된 아이템을 집어넣는다.
    buffer[in] = next_produced;
    
    /* 4. 다음 데이터가 들어갈 인덱스를 갱신한다. */
    // 버퍼의 끝에 도달하면 다시 처음(0)으로 돌아가도록 원형 큐(Circular Queue) 방식을 사용한다.
    in = (in + 1) % BUFFER_SIZE;
    
    /* 5. 버퍼 내 아이템의 총 개수를 1 증가시킨다. */
    counter++; 
}

// Consumer
while (true) {
    /* 1. 버퍼가 비어있는지 검사한다. */
    // counter가 0이라면 꺼낼 아이템이 없으므로, 
    // 생산자가 새로운 아이템을 버퍼에 넣을 때까지 무한 루프를 돌며 대기(Busy Waiting)한다.
    while (counter == 0) ; /* do nothing */
    
    /* 2. 버퍼에서 데이터를 꺼낸다. */
    // 현재 'out'이 가리키는 인덱스 위치의 아이템을 'next_consumed' 변수에 가져온다.
    next_consumed = buffer[out];
    
    /* 3. 다음 데이터를 꺼낼 인덱스를 갱신한다. */
    // 마찬가지로 원형 큐(Circular Queue) 방식을 사용하여 인덱스를 순환시킨다.
    out = (out + 1) % BUFFER_SIZE;
    
    /* 4. 버퍼 내 아이템의 총 개수를 1 감소시킨다. */
    counter--;
    
    /* 5. 꺼낸 아이템('next_consumed')을 실제로 소비(사용)한다. */
}
```
## 코드의 분석 및 한계점 (동기화의 필요성)
### 목적 및 원리:
이 코드는 생산자와 소비자가 하나의 공유 버퍼(Shared Buffer)와 데이터 개수를 추적하는 counter 변수를 사용하여 데이터를 주고받는 논리를 명확하게 보여준다. 생산자는 in 포인터를, 소비자는 out 포인터를 사용하여 각자의 작업 위치를 독립적으로 관리한다.

### 코드의 치명적인 문제점 (경쟁 상태, Race Condition):
위 코드는 단일 프로세스 환경에서는 완벽하게 동작하지만, 프로세스가 동시에(Concurrently) 실행되는 운영체제 환경에서는 데이터 불일치(Data Inconsistency)라는 심각한 오류를 유발한다.

그 이유는 counter++와 counter-- 연산이 고급 언어(C언어 등)에서는 한 줄로 보이지만, 실제 CPU가 처리하는 기계어 수준에서는 다음 세 단계로 나뉘어 실행되기 때문이다.

1. 메모리에서 레지스터로 값을 가져온다 (Load).

2. 레지스터에서 값을 1 증가시키거나 감소시킨다 (Add/Sub).

3. 계산된 레지스터 값을 다시 메모리에 저장한다 (Store).

### 성능 측면에서의 분석:
생산자가 counter++의 3단계 중 2단계까지만 실행한 상태에서 CPU 실행 권한을 빼앗기고(Context Switch), 소비자가 counter--를 실행해버리면 counter 값은 정상적으로 반영되지 않고 덮어씌워지는 현상이 발생한다. 이렇게 여러 프로세스가 공유 데이터에 동시에 접근하여 타이밍에 따라 결과가 달라지는 상황을 경쟁 상태(Race Condition)라고 한다.

시스템의 동시성을 높여 처리 성능을 극대화하기 위해 다중 프로세싱을 사용하지만, 위 코드처럼 공유 변수(counter)에 대한 접근을 통제하지 않으면 데이터 오류를 복구하기 위해 시스템 전체가 정지하거나 치명적인 버그를 수정하기 위해 엄청난 성능적, 시간적 비용을 지불해야 한다. 따라서 이 문제를 해결하기 위해 이후 단계에서 임계구역(Critical Section)을 보호하는 동기화(Synchronization) 기법(예: 뮤텍스, 세마포어)을 반드시 도입해야 한다.
